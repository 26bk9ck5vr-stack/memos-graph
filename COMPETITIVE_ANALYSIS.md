## 🔍 详细对比

### 1. **memos-graph** (本项目)

**定位**: Agent 长期记忆后端引擎

**核心架构**:
```
PostgreSQL (pgvector + pg_jieba)
    ↓
memos-graph API
    ↓
Agent Packs (可插拔)
```

**记忆实现**:
- ✅ **写入**: 实时同步 (35-50ms)，异步向量生成
- ✅ **召回**: 7 阶段混合 (FTS → Vector → Pattern → Time → Graph → MMR → RRF)
- ✅ **中文 FTS**: pg_jieba 集成 (100% 触发率)
- ✅ **向量**: BAAI/bge-m3 1024 维 (SiliconFlow API)
- ✅ **实体抽取**: LLM 自动抽取
- ✅ **事件/承诺追踪**: 自动识别
- ✅ **Pack 系统**: 可插拔 Agent 插件
- ✅ **心跳调度**: 主动消息

**优势**:
1. ✅ **简单**: 单一 PostgreSQL，无额外依赖
2. ✅ **中文优化**: pg_jieba 中文分词
3. ✅ **完整**: 记忆全链路 (写入→召回→注入)
4. ✅ **轻量**: 单体架构，易部署
5. ✅ **诚实**: v1.0.0-beta，明确标注未完成部分

**劣势**:
1. ❌ **无生产验证**: 尚无真实用户长期使用
2. ❌ **生态小**: 无框架集成 (LangChain/AutoGen 等)
3. ❌ **文档少**: 只有核心文档，无教程
4. ❌ **Ollama 未实现**: 只能用 SiliconFlow API

**适用场景**:
- ✅ 需要中文长期记忆的 Agent 项目
- ✅ 希望简单部署 (单一数据库)
- ✅ 需要完整记忆链路 (不只是存储)
- ⚠️ 不适合：需要框架集成 (LangChain 等)

### 2. **LangGraph** (LangChain 出品)

**定位**: 低级别状态机编排框架

**核心架构**:
```
LangGraph State Graph
    ↓
Nodes (LLM + Tools)
    ↓
Edges (条件转移)
    ↓
持久化层 (Redis/Postgres)
```

**记忆实现**:
- ✅ **状态持久化**: Checkpoint 保存/恢复
- ✅ **长时间运行**: 支持中断/恢复
- ✅ **人工介入**: Human-in-the-loop
- ⚠️ **长期记忆**: 依赖外部 (LangChain Memory)
- ❌ **向量搜索**: 无内置 (需集成)
- ❌ **中文优化**: 无特殊优化

**优势**:
1. ✅ **生态强**: LangChain 完整生态
2. ✅ **生产验证**: 大量企业使用
3. ✅ **灵活**: 任意状态图定义
4. ✅ **调试工具**: LangSmith 可视化
5. ✅ **文档全**: 完整教程 + API 参考

**劣势**:
1. ❌ **复杂**: 学习曲线陡峭
2. ❌ **重**: 需要 LangChain 完整栈
3. ❌ **记忆非核心**: 重点是编排，不是记忆
4. ❌ **中文支持弱**: 无中文 FTS

**适用场景**:
- ✅ 复杂 Agent 工作流编排
- ✅ 需要 LangChain 生态
- ✅ 需要人工介入的工作流
- ⚠️ 不适合：只需要简单记忆存储

**劣势**:
1. 🔴 **复杂**: Temporal 依赖 (学习成本高)

3. ✅ **框架集成**: LangChain/AutoGen/ADK 等
4. ✅ **生产验证**: SaaS 用户在使用

**劣势**:
1. 🔴 **依赖云服务**: 必须用 Zep Cloud
2. ❌ **数据出境**: 数据在 Zep 服务器

**适用场景**:
- ✅ 快速原型，不想部署
- ✅ 需要专业记忆功能
- ✅ 不介意数据在云端
- ⚠️ 不适合：数据敏感/需要本地部署
